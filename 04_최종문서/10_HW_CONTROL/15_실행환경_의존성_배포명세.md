# 실행환경·의존성·배포명세

## 1. 버전 관리 원칙

- “latest”를 운영 기준으로 사용하지 않는다.
- 목표 버전과 실장 확인 버전을 분리한다.
- apt·ROS package, Python, container, interface, config, model을 재현 가능한 식별자로 고정한다.
- 보안 업데이트도 HIL·회귀시험 없이 실차 release에 직접 적용하지 않는다.

## 2. Release manifest 필수 필드

| 영역 | 필수 정보 |
|---|---|
| 공통 | release ID, Git SHA, build time, 담당자 |
| Interface | interface repository SHA, type hash |
| Jetson | board, JetPack, L4T, Ubuntu, kernel, ROS, RMW, Python, CUDA, TensorRT, cuDNN |
| RPi | board revision, Ubuntu, kernel, Docker, container digest, ROS, RMW, Python |
| Config | route, calibration, parameters, QoS, config SHA-256 |
| AI | model name/version/artifact hash, preprocessing version |
| HW | wiring revision, motor/driver/servo/encoder identification |

## 3. Dependency 잠금 요구

| 종류 | 잠금 방법 |
|---|---|
| Container | registry digest와 SBOM `[추후 추가 예정]` |
| Python | exact version과 hash가 있는 lock file |
| ROS source | repository commit과 `rosdep` 결과 |
| ROS binary | repository snapshot·package version 기록 |
| Config | Git review와 content hash |
| Model | immutable artifact hash와 dataset·code lineage |

## 4. 배포 요구사항

- Jetson과 RPi는 각각 자신의 `systemd`에서 시작하며 SSH 연쇄 기동을 금지한다.
- replay profile에는 actuator·watchdog device access를 포함하지 않는다.
- 운영 config는 read-only이며 변경 후 drive enable 전 self-test를 수행한다.
- rollback은 이전 manifest·image·config·model을 한 세트로 복원한다.
- rollback이나 restart 뒤에는 수동 승인 전 drive disabled를 유지한다.

## 5. 승격 Gate

`development → bench → HIL → controlled-course → RC` 순으로 승격한다. 각 단계는 동일 release manifest를 사용하고 artifact가 바뀌면 영향을 받는 시험을 다시 수행한다.
