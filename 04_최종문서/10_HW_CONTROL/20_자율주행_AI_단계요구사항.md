# Agent·Teacher 단계요구사항

## 1. 목적

Agent는 camera image에서 normalized steering·throttle 후보를 생성한다. Teacher는 수집 episode에서 Rule·Agent·운영자 결과를 비교하고 failure data와 학습 후보를 만든다.

Agent와 Teacher 모두 actuator hardware를 직접 소유하지 않는다. Agent 후보만 승인된 stage에서 기존 `command_mux → safety_supervisor → actuator_controller` 경로를 통해 제한적으로 적용될 수 있다.

## 2. 입력 기준

차량 runtime의 기본 입력은 camera image·camera info·timestamp·calibration ID다. odometry·IMU·거리 scan·GNSS·route goal은 제공되는 것으로 가정하지 않는다.

따라서 학습 label과 평가는 다음 근거만 사용한다.

- image와 tactile path
- Rule·Agent candidate command
- 실제로 요청한 actuator state
- 운영자의 stop/reset·stage 기록
- 외부 촬영 영상·수동 annotation `[선택]`

## 3. Agent 승격 단계

| 단계 | 실행 | actuator 권한 | Exit Criteria |
|---|---|---:|---|
| `A0 Dataset` | Rule Drive·failure episode 정리 | 0 | episode 분리·calibration·provenance 완전 |
| `A1 Offline` | 개발 환경에서 model 비교·학습 | 0 | held-out episode 결과·artifact hash 생성 |
| `A2 Agent Replay` | Jetson에서 저장 image 추론 | 0 | 같은 입력 재현·latency·resource 기준 통과 |
| `A3 Agent Shadow` | Rule Drive 중 Agent 후보 병행 기록 | Agent 0 | frame별 rule/agent disagreement 수집 |
| `A4 Limited Assist` | 승인 confidence에서 Agent 후보 선택 | 제한적 | limit·fallback·fault injection 통과 |

Public checkpoint나 Offline 점수만으로 `A4`를 열지 않는다. 모든 새 artifact는 `A2 → A3 → A4`를 다시 통과한다.

## 4. Agent Runtime Interface

| 항목 | 계약 |
|---|---|
| 입력 | image rect, camera info, timestamp, calibration ID |
| 출력 | `SOURCE_AGENT` VehicleCommand, `AgentState` |
| Identity | model ID, artifact hash, preprocessing version |
| Health | warm-up, timeout, dropped input, GPU/RAM/temperature |
| Failure | invalid·timeout output 폐기, rule fallback 또는 neutral |

Agent Module은 preprocessing, inference, output normalization, confidence, artifact loading을 작은 ROS Interface 뒤에 숨긴다.

## 5. Shadow·Assist 규칙

### Shadow

- actuator에는 Rule command만 적용한다.
- 같은 frame에서 Rule·Agent 후보와 latency를 기록한다.
- Agent가 stale·invalid여도 Rule 주행에 영향을 주지 않는다.
- disagreement가 큰 frame을 Teacher 후보로 표시한다.

### Limited Assist

- `shadow_only=false`, artifact hash 일치, confidence threshold 이상이어야 한다.
- Agent throttle·steering은 Rule Drive에서 검증한 absolute/slew limit을 넘지 못한다.
- Agent invalid·stale·resource fault 시 Rule fallback 또는 neutral로 전환한다.
- fallback 정책은 stage config에 고정하며 주행 중 변경하지 않는다.
- 장애물 회피나 속도 폐루프 능력이 생긴 것으로 표현하지 않는다.

## 6. Teacher loop

```text
Rule/Agent episode
  → disagreement·stop·lost 구간 추출
  → Teacher 평가·수동 검토
  → failure dataset version
  → Agent 재학습
  → held-out Replay
  → Jetson Agent Replay
  → Shadow
  → 제한 Assist 재승격
```

Teacher는 다음 결과를 만든다.

- frame별 `TeacherEvaluation`
- failure reason과 training 선택 여부
- immutable dataset manifest
- 사용한 evaluator·Teacher model identity
- 이전 Agent 대비 개선·회귀 보고서

Teacher 결과는 자동 정답으로 간주하지 않는다. held-out episode와 운영자 검토 없이 dataset과 model을 승격하지 않는다.

## 7. 데이터 분리

- train/validation/test는 frame 무작위 분할이 아니라 episode·날짜·환경 단위로 나눈다.
- 같은 연속 주행의 인접 frame이 서로 다른 split에 섞이지 않게 한다.
- failure episode는 원본을 보존하고 파생 clip·label·dataset hash를 연결한다.
- Teacher가 사용한 test episode는 다음 학습 train split에 자동 편입하지 않는다.

## 8. 금지 사항

- Agent·Teacher의 `/dev/i2c-*` 접근
- Agent·Teacher의 `/vehicle/target_cmd` 직접 발행
- Replay 결과만으로 Assist 승인
- Teacher 평가만으로 model 자동 배포
- 실제 측정값이 없는 speed·distance·trajectory 성능 주장
