# 최신 자율주행 연구·개발 트렌드 보고서

- **조사 기준일:** 2026-07-16
- **조사 범위:** 2024~2026년 자율주행·모바일 로봇 관련 주요 학회 논문, 최신 사전 공개 논문, Waymo·NVIDIA·ROS 2/Nav2 공식 자료
- **적용 대상:** Jetson Orin Nano와 Raspberry Pi를 사용하는 점자블록·장애물 탐지 자율주행 로봇

---

## 1. Executive Summary

최근 자율주행 연구는 기존의 완전 모듈형 구조에서 **Planning-oriented End-to-End**, **Vision-Language-Action(VLA)**, **World Model**, **생성형 궤적 계획**, **폐루프 평가** 중심으로 빠르게 이동하고 있다. 그러나 실제 시스템에서 하나의 거대 AI 모델이 카메라 입력으로부터 PWM을 직접 출력하는 형태가 주류가 된 것은 아니다.

2025~2026년 연구의 실질적인 공통점은 다음과 같다.

1. 인지 정확도 자체보다 **최종 계획·주행 성능**을 중심으로 모델을 최적화한다.
2. 단일 궤적 회귀보다 **복수의 실행 가능한 궤적 후보**를 생성하고 평가한다.
3. VLM·VLA를 이용해 장면의 의미, 자연어 지시, 장기 의도를 처리한다.
4. 미래 장면을 예측하는 **World Model**을 계획과 시뮬레이션에 연결한다.
5. 느린 의미 추론과 빠른 실시간 제어를 분리하는 **Dual-System 구조**가 중요해지고 있다.
6. 학습 모델의 출력은 독립된 **Safety Supervisor 또는 Runtime Assurance 계층**을 통과한다.
7. 녹화 데이터 기반 Open-loop 점수보다 실제 행동 결과를 반영하는 **Closed-loop 평가**가 중시된다.
8. 차량 탑재 모델은 거대 교사 모델을 그대로 사용하는 대신 **경량화·증류·비동기 실행**하는 방향으로 개발된다.

본 프로젝트에는 다음 구조가 가장 적합하다.

```text
Jetson
├─ Perception
├─ World State / Local Occupancy
├─ Behavior Tree 기반 자율주행 에이전트
├─ Fast Local Planner & Path Tracker
├─ 필요 시 실행되는 Slow Semantic Agent
└─ Safety Supervisor
        ↓ 목표 속도·목표 조향각
Raspberry Pi
├─ 엔코더 기반 속도 PID
├─ 서보 PWM 제어
├─ Watchdog
└─ 하드웨어 비상정지
```

즉, 최신 기술을 반영하되 **VLM이 직접 모터를 제어하는 구조가 아니라, Jetson 에이전트가 행동 의도와 기준 궤적을 결정하고 결정론적 제어기와 안전 계층이 최종 명령을 생성하는 구조**를 권장한다.

## 3. 자율주행 아키텍처의 변화

### 3.1 전통적 모듈형 구조

```text
센서
→ 객체·차선 인지
→ 객체 추적
→ 행동 예측
→ 행동 계획
→ 경로 계획
→ 차량 제어
```

장점은 각 모듈을 개별적으로 검증할 수 있고 오류 원인을 추적하기 쉽다는 것이다. 반면 모듈 사이의 수작업 인터페이스와 누적 오차가 전체 성능을 제한한다.

### 3.2 Planning-oriented End-to-End

최근의 End-to-End 자율주행은 대체로 센서 입력으로부터 최종 **미래 궤적 또는 waypoint**를 학습한다.

```text
카메라·LiDAR·차량 상태
→ 통합 특징 표현
→ 미래 궤적·행동 후보
→ 제어기
```

여기서 End-to-End는 반드시 카메라에서 PWM을 직접 생성한다는 의미가 아니다. 최신 시스템은 다음 중 하나를 출력하는 경우가 많다.

- 미래 궤적 좌표
- waypoint 집합
- 복수 궤적과 각 후보의 점수
- 행동 의도와 제한 조건
- 목표 속도·곡률

Waymo의 EMMA는 멀티모달 언어 모델을 기반으로 카메라 입력에서 계획 궤적, 객체, 도로 그래프를 통합 생성한다. 하지만 적은 수의 프레임만 처리하고 LiDAR·Radar를 사용하지 않으며 계산 비용이 크다는 한계도 공식적으로 제시했다.[2]

### 3.3 최근의 하이브리드 구조

실제 개발에서 가장 현실적인 형태는 다음과 같다.

```text
학습 기반 인지·추론
+
결정론적 계획·제어
+
독립 안전 감시
```

이 방식은 학습 모델의 일반화 능력을 사용하면서도 실시간성과 안전성, 디버깅 가능성을 유지한다.

## 4. 핵심 연구·개발 트렌드

## 4.1 Vision-Language-Action(VLA)

VLA는 영상 인지, 언어 기반 추론, 행동 출력을 하나의 모델 또는 연결된 모델군으로 처리한다.

```text
Vision
- 카메라 영상
- BEV·3D 표현

Language
- 주행 지시
- 교통 규칙
- 상황 설명
- 원인과 결과 추론

Action
- 행동 의도
- waypoint
- 궤적
- 속도·조향 목표
```

VLA 연구는 다음 두 계열로 구분된다.[3][4]

### End-to-End VLA

하나의 모델이 영상·언어 입력에서 행동 또는 궤적까지 직접 생성한다.

**장점**

- 인지와 행동을 공동 최적화할 수 있다.
- 자연어 지시와 일반 지식을 주행 판단에 연결할 수 있다.
- 복잡한 장면에 대한 설명을 함께 생성할 수 있다.

**한계**

- 추론 지연이 크다.
- 공간·거리 정밀도가 부족할 수 있다.
- 동일 입력에 대한 출력 일관성 보장이 어렵다.
- 물리 제약과 안전 제약을 형식적으로 검증하기 어렵다.

### Dual-System VLA

VLM이 느린 고수준 판단을 담당하고, 별도의 경량 주행 모듈이 빠른 제어를 담당한다.

```text
Slow System
VLM·VLA
- 의미 이해
- 장기 의도
- 행동 후보
        ↓
Fast System
Local Planner·Controller
- 장애물 반응
- 궤적 계산
- 속도·조향 출력
```

2025년 말 VLA 조사에서는 현재 방법을 End-to-End VLA와 Dual-System VLA로 분류하며, Dual-System은 VLM의 고수준 추론과 전문 주행 모듈의 저지연 실행을 분리한다고 정리한다.[4]

### 산업 방향

NVIDIA는 2026년 Alpamayo 계열을 장기 꼬리 상황에 대한 reasoning VLA, 시뮬레이션, 데이터셋 생태계로 발표했다. 중요한 점은 대형 모델을 차량에서 그대로 구동하기보다 **교사 모델로 사용하고 소형 런타임 모델로 미세조정·증류**하도록 제안한다는 것이다.[5]

### 프로젝트 적용 판단

- VLA를 10~30 Hz의 직접 조향 제어기로 사용: **비권장**
- VLA를 행동 모드·복구 전략·고수준 waypoint 생성기로 사용: **권장**
- VLA의 결과를 결정론적 Planner와 Safety Supervisor로 검증: **필수**

## 4.2 World Model

World Model은 현재 상태와 후보 행동으로부터 미래 장면 또는 미래 잠재 상태를 예측한다.

```text
현재 관측 + 과거 관측 + 후보 행동
→ 미래 환경 상태 예측
→ 충돌·진행도·경로 일치 평가
→ 행동 선택
```

### 주요 유형

| 유형 | 출력 | 장점 | 한계 |
|---|---|---|---|
| 생성형 영상 World Model | 미래 카메라·LiDAR | 사람이 결과를 확인하기 쉬움 | 연산량이 크고 시각적 사실성이 물리 정확성을 보장하지 않음 |
| Latent World Model | 압축된 미래 특징 | 계획 모듈과 연결하기 쉬움 | 해석이 어렵고 학습 품질 검증 필요 |
| Occupancy World Model | 미래 점유 공간 | 충돌 및 주행 가능 영역 평가에 적합 | 정확한 3D 데이터와 학습 비용 필요 |

CVPR 2026의 DriveLaW는 미래 영상 생성기의 잠재 표현을 diffusion planner에 직접 전달해 World Prediction과 Motion Planning을 통합했다.[6] ICCV 2025의 World4Drive도 물리적 잠재 World Model을 사용해 복수 궤적을 생성하고 평가한다.[7]

World Model의 성능은 영상이 사실적으로 보이는지만으로 평가할 수 없다. CVPR 2026 WorldLens는 생성 품질, 재구성, 행동 추종, 다운스트림 작업, 사람 선호를 포함하는 다면적 평가를 제안했으며, 단일 모델이 모든 평가축에서 우수하지 않다고 보고한다.[8]

### 산업 적용

Waymo는 2026년 카메라와 LiDAR를 생성하고 언어·행동·장면 배치로 제어할 수 있는 World Model 기반 시뮬레이션을 공개했다. 핵심 용도는 실제 도로에서 수집하기 어려운 희귀·장기 꼬리 시나리오의 생성과 검증이다.[9]

NVIDIA Cosmos 역시 World Foundation Model을 물리 AI 시뮬레이션, 미래 예측, 합성 데이터 생성에 사용한다.[10]

### 프로젝트 적용 판단

현재 프로젝트에서 World Model을 직접 주행의 핵심으로 사용하기보다 다음 순서가 타당하다.

1. 장애물·점자블록·로봇 footprint 기반 Local Occupancy 구축
2. 0.5~2초의 짧은 미래 점유 예측
3. 후보 회피 경로의 충돌 여부 평가
4. 이후 Latent World Model 연구로 확장

## 4.3 생성형 궤적 계획: Diffusion과 Flow Matching

단일 정답 궤적 회귀는 복수의 안전한 선택지가 존재하는 장면에서 평균적이고 부자연스러운 궤적을 생성할 수 있다. 최근에는 궤적 분포를 모델링하는 생성형 Planner가 증가하고 있다.

```text
장면 표현
→ 직진·좌회피·우회피·감속 등 복수 후보 생성
→ 안전성·진행도·편안함·경로 일치 평가
→ 최종 궤적 선택
```

### DiffusionDrive

CVPR 2025 DiffusionDrive는 사전 정의한 multi-mode anchor와 truncated diffusion을 사용해 denoising 단계를 2단계로 줄였다. 논문은 기존 diffusion policy 대비 denoising 횟수를 10배 줄이고, RTX 4090에서 45 FPS를 보고했다.[11]

### GuideFlow

CVPR 2026 GuideFlow는 Flow Matching 과정에 명시적인 안전·주행 스타일 제약을 주입한다. 생성형 Planner가 다양한 후보를 생성하면서도 hard constraint를 만족하도록 유도하는 방향을 제시한다.[12]

### 트렌드 해석

단순히 “생성형 모델을 사용한다”는 것보다 다음 변화가 중요하다.

- 단일 궤적이 아니라 다중 후보를 생성한다.
- 후보 생성 과정에 속도·차선·충돌 제약을 직접 넣는다.
- 생성 모델의 무작위성을 별도 평가기 또는 안전 필터로 제어한다.
- 실시간 실행을 위해 step 수를 줄이고 증류한다.

### 프로젝트 적용 판단

Jetson Orin Nano MVP에서 diffusion planner를 바로 핵심 기능으로 채택하는 것은 부담이 크다. 다음 단계적 적용이 적절하다.

```text
규칙 기반 회피 후보 3개 생성
→ 결정론적 비용함수로 평가
→ 데이터 축적
→ 경량 학습 기반 후보 평가기
→ 필요 시 Diffusion·Flow Planner
```

## 4.4 시간·메모리 기반 계획

단일 프레임 기반 판단은 순간 가림, 조명 변화, 점자블록 단절에서 흔들릴 수 있다. 최근 연구는 이전 인지 결과와 이전 계획을 현재 판단에 포함한다.

CVPR 2025 MomAD는 trajectory momentum과 perception momentum을 활용해 시간적 일관성을 높이고 steering instability와 단일 프레임 가림 문제를 줄이는 방향을 제안했다.[13]

### 프로젝트 적용 예

```text
현재 프레임 점자블록 미검출
→ 즉시 완전 정지하지 않음
→ 최근 중심선 + IMU Yaw + Encoder 이동량으로 짧게 예측
→ 재검출 시 정상 복귀
→ 일정 시간 이상 실패 시 GPS 경로 또는 안전 정지
```

필요한 구현 요소는 다음과 같다.

- 최근 N프레임 중심선 버퍼
- 중심선 confidence의 이동평균
- 이전 조향각과 곡률의 변화율 제한
- 객체·경로 tracking ID
- 이벤트 중복 방지 메모리
- 회피 전 원래 오프셋 경로 저장

## 4.5 BEV·3D Geometry·Occupancy 중심 표현

최근 자율주행은 객체 Bounding Box 목록보다 **어느 공간이 점유되어 있고 어느 공간이 주행 가능한가**를 표현하는 방향으로 이동하고 있다.

```text
센서 데이터
→ BEV / Occupancy / Costmap
→ 로봇 footprint와 충돌 검사
→ Local Planning
```

주요 표현은 다음과 같다.

- Bird's-eye View
- 2D/3D Occupancy Grid
- Drivable Area
- Free Space
- Dynamic Occupancy
- Future Occupancy
- 3D Point Map

### 프로젝트 적용

점자블록 로봇은 차체뿐 아니라 측면 카메라 브래킷이 돌출될 수 있다. 따라서 costmap의 footprint는 다음 전체 외형을 포함해야 한다.

```text
차체 폭
+ 바퀴·범퍼
+ 측면 카메라 암
+ 안전 여유 거리
```

LiDAR의 최소 거리 한 개만 사용하는 것보다 다음 구조가 안정적이다.

```text
LaserScan
→ Local Occupancy Grid
→ Inflation
→ 전체 footprint 충돌 검사
→ 좌·우 회피 후보 평가
```

## 4.6 Agentic Autonomy와 Multi-Planner Scheduling

최근 연구에서 “에이전트”는 하나의 모델이 모든 연산을 수행한다는 의미보다, 상황에 따라 여러 전문 Planner와 복구 행동을 선택·조정하는 **Executive 계층**의 의미로 사용되는 경우가 많다.

CVPR 2026의 Open-Ended Instruction Realization 연구는 LLM이 자연어 지시를 해석하고 실행 가능한 스크립트를 생성해 여러 MPC 기반 Planner를 스케줄링하도록 구성한다. 이 구조는 의미 추론과 차량 제어를 서로 다른 시간 척도로 분리해 추적 가능성을 높인다.[14]

ROS 2 Nav2도 Behavior Tree를 사용해 독립된 Planner, Controller, Behavior 서버를 오케스트레이션한다. Nav2 공식 문서는 복잡한 상태와 전이가 증가할 때 Behavior Tree가 FSM보다 확장성과 가독성이 높다고 설명한다.[15]

### 프로젝트에서의 Agent 정의

```text
Autonomy Agent
├─ 현재 World State 읽기
├─ 주행 모드 결정
├─ 사용할 Planner 선택
├─ 이벤트 처리 순서 관리
├─ 실패 시 Recovery Subtree 실행
└─ Safety Supervisor에 후보 명령 전달
```

### 권장 행동 트리

```text
ROOT
└─ SafetyFallback
   ├─ EmergencySequence
   │  ├─ IsEmergency?
   │  └─ EmergencyStop
   └─ MissionSequence
      ├─ CheckSystemHealth
      └─ DrivingFallback
         ├─ ObstacleSequence
         │  ├─ IsObstacleDangerous?
         │  ├─ StopVehicle
         │  ├─ CaptureEvent
         │  ├─ PlanAvoidance
         │  ├─ ExecuteAvoidance
         │  └─ RecoverOffsetPath
         ├─ PathLostSequence
         │  ├─ IsTactilePathLost?
         │  ├─ FollowGlobalRoute
         │  ├─ SearchTactilePath
         │  └─ RecoverOffsetPath
         └─ NormalDriveSequence
            ├─ GenerateOffsetPath
            ├─ TrackPath
            └─ PublishCommand
```

## 4.7 Runtime Assurance와 Safety Supervisor

학습 모델이 생성한 명령은 차량 또는 로봇에 바로 전달하지 않고, 별도의 안전 계층이 검사해야 한다.

```text
에이전트 후보 명령
→ 명령 timestamp와 유효기간 검사
→ 센서·통신 상태 확인
→ 속도·조향 제한
→ 정지거리 검사
→ footprint 충돌 예측
→ 안전하면 승인
→ 위험하면 감속·정지 명령으로 대체
```

2026년 Mission-Level Runtime Assurance 연구는 즉시 충돌하지 않는 명령이라도 체크포인트 누락, 금지구역 진입, 향후 미션 수행 불가와 같은 상위 계획 오류가 존재할 수 있음을 지적하고, 실행 전에 고수준 명령을 거부하는 계층형 감시 구조를 제안한다.[16]

### 프로젝트의 Safety Supervisor 규칙

- LiDAR 또는 핵심 인지 노드 timeout 시 정지
- Jetson→RPi 명령이 0.5초 이상 갱신되지 않으면 정지
- 목표 조향각·속도 제한
- 조향 변화율 제한
- 장애물 거리 기반 최대 속도 제한
- 카메라 confidence 하락 시 감속
- footprint 충돌 예상 시 에이전트 명령 무시
- RPi feedback fault 시 Emergency Stop
- 비상정지는 AI·ROS 프로세스와 독립적으로 모터 전원 차단

## 4.8 Closed-loop 평가와 Fault Injection

### Open-loop 평가

```text
녹화된 센서 입력
→ 모델 궤적 예측
→ 정답 궤적과 거리 비교
```

계산이 빠르지만 모델 행동이 다음 입력을 변화시키는 누적 오차를 반영하지 못한다.

### Closed-loop 평가

```text
모델 행동
→ 로봇 위치·방향 변경
→ 변경된 상태에서 새 센서 입력
→ 다시 판단
```

Bench2Drive는 44개 상호작용 시나리오, 다양한 날씨·지역을 포함한 폐루프 평가를 제안했다.[17] NAVSIM은 실제 데이터 기반 평가와 시뮬레이션 기반 지표 사이의 중간 접근을 제공하며, 진행도와 충돌 가능성 등 계획 중심 지표를 사용한다.[18]

2026년 Open-loop와 Closed-loop의 상관관계를 비교한 연구는 최신 안전 지표도 폐루프 순위를 완전히 보장하지 않으며, 작은 편차가 누적되는 snowball effect가 남는다고 보고한다.[19]

### 프로젝트 평가에 포함할 Fault Injection

- 전방 카메라 0.2~1초 지연
- 프레임 누락
- 점자블록 segmentation confidence 급락
- LiDAR 일부 각도 누락
- GPS 위치 편차
- IMU yaw bias
- 엔코더 한쪽 오차
- Jetson 추론 지연 증가
- Ethernet 패킷 지연·손실
- RPi feedback timeout
- 장애물 회피 중 점자블록 미검출
- 회피 후 원래 경로와 반대 방향 재진입

### 권장 평가 지표

| 영역 | 지표 |
|---|---|
| 경로 추종 | 평균·최대 횡오차, heading error, 곡선 추종 성공률 |
| 안정성 | 조향 변화율, jerk, 좌우 흔들림 횟수 |
| 장애물 | 정지거리, 충돌률, 최소 clearance, 회피 성공률 |
| 복구 | 점자블록 재검출 시간, 오프셋 경로 복귀 시간·성공률 |
| 시스템 | 주기별 latency, deadline miss, timeout 횟수 |
| AI | 결함 검출률, 오탐률, 이벤트 중복률 |
| 미션 | 전체 루트 완주율, 중단 원인, 구간별 진행도 |

## 6.2 Slow Semantic Agent의 실행 조건

VLM 또는 소형 VLA는 상시 제어 루프가 아니라 다음 이벤트에서만 호출한다.

- 경로 후보가 두 개 이상이고 GPS 정보만으로 선택하기 어려운 경우
- 장애물의 의미에 따라 정지·대기·회피 판단이 필요한 경우
- 점자블록 재탐색이 반복 실패한 경우
- 관리자의 자연어 미션을 행동 명령으로 변환하는 경우
- 이상 상황의 설명과 로그 요약이 필요한 경우

정상 주행에서는 Fast Planner만 실행한다.

```text
정상 상태
→ 결정론적 Fast Planner

불확실성 임계값 초과
→ 감속 또는 정지
→ Slow Semantic Agent 호출
→ 행동 후보 생성
→ Safety Supervisor 검증
```

## 6.3 Jetson에서 RPi로 전달할 명령

Jetson은 PWM이 아닌 물리적 목표값을 전달한다.

```yaml
stamp: 2026-07-16T10:30:00.123+09:00
sequence: 1524
mode: NORMAL
source: OFFSET_PATH_TRACKER

target_speed_mps: 0.20
target_steering_rad: -0.14

limits:
  max_speed_mps: 0.25
  max_steering_rad: 0.35

confidence: 0.91
valid_for_ms: 200
emergency_stop: false
```

RPi는 다음을 수행한다.

```text
target_speed_mps
→ Encoder PID
→ Motor PWM

target_steering_rad
→ 조향 범위 제한
→ 변화율 제한
→ PWM 변환
→ Steering Servo
```

## 9. 피해야 할 설계

### 9.1 VLM에서 PWM 직접 출력

```text
Camera → VLM → Motor PWM
```

문제점:

- 추론 지연이 제어 주기와 직접 결합된다.
- 명령의 물리적 타당성을 검증하기 어렵다.
- 모델 오류가 즉시 하드웨어 동작으로 이어진다.
- 재현성과 디버깅이 어렵다.

### 9.2 Jetson 통신 단절 시 마지막 명령 유지

RPi는 명령 유효기간이 끝나면 즉시 속도 목표를 0으로 전환해야 한다.

### 9.3 GPS를 횡방향 정밀 제어에 사용

GPS는 글로벌 루트와 이벤트 위치에 사용하고, cm 단위 오프셋 제어는 카메라·IMU·엔코더·LiDAR로 처리해야 한다.

### 9.4 YOLO mAP만으로 작품 성능 평가

자율주행 작품의 핵심은 검출률뿐 아니라 폐루프 완주율, 충돌률, 경로 오차, 회피·복구 성공률이다.

### 9.5 대형 연구 모델을 Jetson에 그대로 탑재

대형 VLA·World Model은 먼저 서버 또는 오프라인 Teacher로 사용하고, Jetson에는 경량화·증류된 모델 또는 이벤트 기반 호출 구조를 적용해야 한다.

## 10. 최종 제안

본 프로젝트가 최신 자율주행 연구 트렌드를 반영하면서도 실제 구현 가능성을 유지하려면 다음 아키텍처를 채택하는 것이 적절하다.

> **Dual-System Agentic Autonomous Driving Architecture**
>
> Jetson의 자율주행 에이전트가 카메라·LiDAR·GPS·IMU·엔코더 정보를 World State로 통합하고, Behavior Tree를 통해 주행 모드와 Planner를 선택한다. Fast Planner가 실시간 궤적과 목표 속도·조향각을 생성하며, Slow Semantic Agent는 불확실하거나 복잡한 이벤트에서만 동작한다. 모든 후보 명령은 독립 Safety Supervisor를 통과한 뒤 Raspberry Pi로 전달되고, Raspberry Pi는 모터 PID·조향 PWM·Watchdog·하드웨어 비상정지를 수행한다.

최신 연구의 핵심을 프로젝트 관점에서 압축하면 다음과 같다.

```text
Foundation Model을 직접 제어기로 사용
             X

Foundation Model로 의미·장기 판단을 강화
+ 경량 Fast Planner로 실시간 실행
+ Safety Supervisor로 최종 검증
             O
```

따라서 우선 개발해야 하는 것은 거대 VLA 자체가 아니라 다음 다섯 요소다.

1. **World State와 Short-term Memory**
2. **Behavior Tree 기반 Autonomy Agent**
3. **Local Occupancy와 Planner Scheduling**
4. **독립 Safety Supervisor**
5. **Closed-loop·Fault Injection 평가 체계**

이 기반 위에 VLA, World Model, Diffusion Planner를 순차적으로 추가해야 연구성·안전성·구현 가능성을 동시에 확보할 수 있다.

## 부록 C. Safety Supervisor 의사코드

```python
def supervise(command, world_state):
    if world_state.health.emergency_stop_active:
        return stop_command(reason="physical_estop")

    if not world_state.health.rpi_connected:
        return stop_command(reason="rpi_disconnected")

    if not world_state.health.lidar_ok:
        return stop_command(reason="lidar_timeout")

    if command.is_expired():
        return stop_command(reason="command_timeout")

    safe_speed = min(
        command.target_speed_mps,
        speed_limit_from_curvature(world_state.path.curvature),
        speed_limit_from_confidence(world_state.path.confidence),
        speed_limit_from_obstacle(world_state.obstacle.min_range_m),
    )

    safe_steering = clamp_rate_and_range(
        command.target_steering_rad,
        previous_steering,
    )

    if footprint_collision_predicted(safe_speed, safe_steering):
        return stop_command(reason="predicted_collision")

    return command.with_values(
        target_speed_mps=safe_speed,
        target_steering_rad=safe_steering,
    )
```
