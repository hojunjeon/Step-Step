# 점자블록 불량·장애물 탐지 시스템 예상 문제 및 해결방안

## 1. 긴 크랙이 여러 객체로 분할 탐지되는 문제

**문제:**  
하나의 긴 크랙이 YOLO의 여러 바운딩 박스로 분할될 수 있다. 이 경우 동일한 크랙이 지도에 여러 개의 결함으로 표시되고, 해당 구간의 불량 개수와 보수 물량이 과대 산정될 수 있다.

**가장 가능성 높은 해결방안:**  
단순 객체 탐지 모델보다 YOLO Segmentation 계열 모델을 사용해 크랙의 실제 영역을 마스크 형태로 추출한다.

여러 크랙 마스크가 다음 조건을 만족하면 하나의 결함으로 병합한다.

- 같은 프레임 또는 연속 프레임에서 검출
- 마스크 사이 거리가 일정 기준 이하
- 크랙 진행 방향이 유사
- 동일한 점자블록 또는 인접 점자블록에 위치
- 로봇 진행 방향 기준으로 공간적으로 이어짐

처리 흐름:

```text
여러 크랙 검출
→ Segmentation 마스크 생성
→ 근접도·방향·연결성 검사
→ 하나의 크랙 이벤트로 병합
→ 전체 길이와 면적 산출
```

## 2. 동일 불량이 크랙과 파손으로 번갈아 분류되는 문제

**문제:**  
동일한 결함이 프레임별로 `크랙 → 파손 → 크랙`처럼 분류될 수 있다. 이를 그대로 서버에 전송하면 하나의 결함에 서로 다른 아이콘이 생성되거나 시점별 상태 변화가 잘못 기록될 수 있다.

**가장 가능성 높은 해결방안:**  
한 프레임의 결과를 즉시 확정하지 않고, 객체가 화면 중앙에 진입하기 전후의 연속 프레임 결과를 모아 다수결 또는 신뢰도 누적 방식으로 최종 클래스를 결정한다.

예시:

```text
최근 10개 프레임 기준
- 크랙: 7회, 평균 신뢰도 0.84
- 파손: 3회, 평균 신뢰도 0.68
→ 최종 분류: 크랙
```

추가로 크랙과 파손이 모두 높은 확률로 검출되면 다음처럼 복수 후보를 전송한다.

```json
{
  "primary_class": "crack",
  "secondary_class": "damage",
  "needs_server_review": true
}
```

서버 VLM은 촬영 이미지와 YOLO 결과를 함께 받아 최종 분류를 보정한다.

## 3. 실제로 크랙과 파손이 동시에 존재하는 복합 결함

**문제:**  
동일한 점자블록에서 크랙과 파손이 실제로 동시에 발생할 수 있다. 이를 단일 클래스 문제로 학습시키면 모델이 어느 한쪽을 오분류한 것처럼 보일 수 있다.

**가장 가능성 높은 해결방안:**  
데이터 라벨링을 단일 클래스가 아닌 다중 라벨 방식으로 구성한다.

```text
현재 방식:
한 이미지 = 크랙 또는 파손

권장 방식:
한 이미지 = 크랙 + 파손 가능
```

지도 아이콘은 대표 결함 하나만 표시하되, 상세 화면에서는 복합 결함을 모두 보여준다. 대표 아이콘은 위험도가 가장 높은 클래스를 기준으로 선택한다.

## 4. 정상적인 점자블록 이음새를 크랙으로 인식하는 문제

**문제:**  
블록 사이의 줄눈, 시공 이음부, 블록 테두리, 표면 무늬가 크랙과 유사하게 보일 수 있다. 카메라가 비스듬한 각도로 촬영하면 정상 이음새도 불규칙한 균열처럼 나타날 수 있다.

**가장 가능성 높은 해결방안:**  
학습 데이터에 다음과 같은 하드 네거티브 데이터를 충분히 포함한다.

- 정상적인 블록 이음새
- 시멘트 줄눈
- 블록 테두리
- 물자국과 얼룩
- 그림자
- 낙엽이나 가는 나뭇가지
- 정상적인 돌기 사이 홈

추가로 먼저 점자블록 영역을 segmentation하고, 그 내부에서만 결함을 탐지하도록 2단계 구조를 적용한다.

```text
1단계: 점자블록 영역 분할
2단계: 점자블록 내부의 크랙·파손 탐지
```

## 5. 하나의 점자블록 불량이 여러 프레임에서 반복 촬영되는 문제

**문제:**  
객체가 화면 중앙 경계에서 흔들리거나 로봇이 저속으로 이동하면 촬영 조건이 여러 차례 만족될 수 있다.

**가장 가능성 높은 해결방안:**  
객체별 상태를 추적하는 트래킹 기반 1회 촬영 방식을 사용한다.

```text
DETECTED
→ APPROACHING_CENTER
→ CAPTURED
→ PASSED
```

한 번 `CAPTURED` 상태가 된 객체는 화면에서 완전히 사라질 때까지 다시 촬영하지 않는다. ByteTrack, BoT-SORT 또는 간단한 중심점·IoU 기반 추적을 사용할 수 있다.

## 6. 긴 결함이 화면 중앙에 들어오는 부분마다 다시 촬영되는 문제

**문제:**  
긴 크랙은 첫 부분이 화면 중앙을 지난 후에도 나머지 부분이 계속 새로운 객체처럼 중앙에 진입할 수 있다.

**가장 가능성 높은 해결방안:**  
개별 바운딩 박스가 아니라 결함 구간 이벤트를 생성한다.

```text
크랙 최초 검출
→ 크랙 구간 시작

연속 검출 유지
→ 프레임과 마스크 누적

일정 시간 동안 크랙 미검출
→ 크랙 구간 종료

대표 이미지·전체 길이·최대 폭·면적 저장
```

서버에는 개별 프레임이 아니라 하나의 구간 단위 결과를 전송한다.

## 7. 카메라 거리와 각도에 따라 위험도가 달라지는 문제

**문제:**  
같은 크랙이라도 가까이 촬영하면 크게 보이고 멀리 촬영하면 작게 보인다. 이미지 픽셀 면적만으로 위험도를 계산하면 촬영 조건에 따라 결과가 달라진다.

**가장 가능성 높은 해결방안:**  
카메라 높이와 각도를 고정하고, 카메라 캘리브레이션을 통해 이미지 픽셀을 실제 길이 단위로 변환한다.

- 크랙 길이: cm
- 크랙 최대 폭: mm 또는 cm
- 파손 면적: cm²
- 벗겨짐 비율: 해당 블록 면적의 %
- 단차: mm

LiDAR나 깊이 카메라를 결합하면 거리 보정과 실제 크기 산출이 더 안정적이다.

## 8. 햇빛, 그림자, 야간, 비에 따른 오탐 문제

**문제:**  
직사광선, 나무 그림자, 젖은 표면, 반사광, 야간 조명은 크랙이나 변색처럼 보일 수 있다.

**가장 가능성 높은 해결방안:**  

- 다양한 날씨와 시간대의 학습 데이터 확보
- 자동 노출과 화이트밸런스 고정 또는 기록
- 흔들림·과노출·저조도 이미지 품질 검사
- 품질이 낮으면 조금 이동한 후 재촬영
- 동일 구간을 여러 순찰에서 관찰하고 일회성 이상값 제거
- 서버에 날씨, 조도, 촬영 품질 점수 저장

---

## 9. 물, 먼지, 낙엽 등을 파손이나 벗겨짐으로 오인하는 문제

**문제:**  
흙, 눈, 물웅덩이, 낙엽, 테이프 등이 점자블록 표면을 가리면 벗겨짐 또는 파손으로 오인할 수 있다.

**가장 가능성 높은 해결방안:**  
한 번의 순찰 결과로 영구 결함을 확정하지 않고, 다음 순찰에서도 같은 구간에서 반복 검출되는지를 확인한다.

```text
1회 검출: 잠정 결함
2회 연속 검출: 결함 가능성 높음
3회 이상 반복: 유지보수 후보 확정
```

위험도가 매우 높은 파손이나 단차는 첫 검출부터 우선 검토 대상으로 표시한다.

## 10. 서로 다른 재질과 제조사의 점자블록을 불량으로 오인하는 문제

**문제:**  
고무, 콘크리트, 석재, 금속, 세라믹 등 재질에 따라 표면 질감과 색상이 다르다. 새 블록과 오래된 블록의 제조 형상 차이도 불량으로 오인될 수 있다.

**가장 가능성 높은 해결방안:**  

- 지역과 재질별 데이터셋 구성
- 촬영 구간의 재질을 별도 메타데이터로 저장
- VLM이 재질과 결함을 구분하도록 학습
- 신규 지역 투입 전 소량의 현장 데이터로 추가 학습

## 11. 한 블록의 부분 파손과 여러 블록의 구간 파손을 동일하게 처리하는 문제

**문제:**  
한 장의 일부가 파손된 경우와 여러 미터 구간에 걸쳐 여러 블록이 손상된 경우를 동일한 `파손 1건`으로 처리하면 실제 보수 물량을 알 수 없다.

**가장 가능성 높은 해결방안:**  
탐지 건수보다 영향 구간 길이와 손상 블록 수를 주요 지표로 사용한다.

```text
불량 건수: 8건
손상 블록 추정 수: 23장
영향 구간: 14.2m
심각도 높은 구간: 4.7m
```

## 12. 단차를 RGB 이미지 분류만으로 판단하기 어려운 문제

**문제:**  
단차는 외관상 그림자나 이음새와 유사하며, 실제 높이 차이를 평면 이미지로 정확히 측정하기 어렵다.

**가장 가능성 높은 해결방안:**  
YOLO는 단차 후보 영역을 탐지하고, 최종 단차 여부와 높이는 LiDAR 또는 깊이 센서로 확인한다.

```text
카메라: 단차 후보 검출
LiDAR/Depth: 높이 차이 측정
서버: 단차 위험도 산출
```

## 13. 분류 성능은 높지만 보수 관점에서는 부정확한 문제

**문제:**  
모델이 `크랙`을 정확히 분류하더라도, 미세한 크랙과 구조적 파손으로 진행 중인 크랙을 같은 수준으로 처리하면 보수 계획에 활용하기 어렵다.

**가장 가능성 높은 해결방안:**  
YOLO의 역할을 결함 종류 식별로 제한하고, 서버 VLM이 다음 속성을 별도로 산출한다.

- 결함 면적
- 진행성 가능성
- 보행 기능 저하 정도
- 돌기 손실 비율
- 블록 교체 필요 여부
- 부분 보수 가능 여부
- 관찰 유지 여부

즉, `클래스`와 `보수 판단`을 분리한다.

## 권장 우선순위

### 1순위

- 긴 크랙의 구간 단위 병합
- 연속 프레임 기반 최종 클래스 확정
- 객체 추적 기반 1회 촬영
- 정상 이음새와 결함 구분
- 단차의 LiDAR/Depth 검증

### 2순위

- 복합 결함 다중 라벨
- 실제 크기 단위 보정
- 일시적 오염물과 영구 결함 구분
- 재질별 모델 보정

### 3순위

- 구간별 손상 블록 수 추정
- VLM 기반 보수 판단
- 반복 순찰 기반 결함 신뢰도 누적
