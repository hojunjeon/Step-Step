# AI 학습 및 추론 설계

## 0. AI 시스템의 역할

AI는 세 가지 서로 다른 역할을 수행한다.

1. **주행용 점자블록 인식**
   - 선형·점형블록 영역과 진행 방향을 빠르게 추출
   - 10 FPS 이상을 우선
2. **불량 점자블록 탐지**
   - 마모·벗겨짐, 크랙, 파손, 단차 후보를 segmentation
   - 대표 프레임과 결함 영역 산출
3. **서버 상세 분석**
   - VLM과 규칙이 결함 정도·위험도·추천 조치·교체 시기 범주를 생성
   - 로봇 제어와 분리된 비동기 처리

주행 모델과 불량 모델을 하나로 합치지 않는다. 주행은 모든 프레임에서 안정적으로 동작해야 하지만 불량 탐지는 상대적으로 낮은 주기로 실행해도 되며, 데이터 특성과 최적화 목표도 다르기 때문이다.

## 1. 모델 구성

## 1.1 모델 A: 점자블록 주행 Segmentation

### 클래스

| 클래스 | 의미 |
|---|---|
| `LINEAR_TACTILE` | 진행 방향을 제공하는 선형블록 |
| `WARNING_TACTILE` | 정지·분기·위험 위치를 알리는 점형블록 |

### 출력 활용

- 점자블록 전체 마스크
- 선형블록 중심선
- 가지 방향
- 점형블록 위치
- 마스크 신뢰도
- 시간적 연속성

### 권장 모델

- YOLOv8n-seg 또는 동급 경량 segmentation
- 입력 640×384 또는 640×640
- ONNX/TensorRT 배포
- 목표 추론 10~15 FPS

색상 threshold만으로 시작할 수 있으나, 노란색이 아닌 재질·그림자·변색·젖은 표면에서 실패할 수 있으므로 최종 주행은 학습 기반 segmentation을 사용한다. 색상·기하 규칙은 모델 실패 시 보조 신뢰도로만 사용한다.

## 1.2 모델 B: 불량 Segmentation

### 평가 MVP 클래스

| 내부 클래스 | 한글 | 라벨 정의 |
|---|---|---|
| `SURFACE_WEAR` | 마모·벗겨짐 | 표면 박리, 돌기 일부 손실, 넓은 마모 |
| `CRACK` | 크랙 | 선형 또는 망상 균열 |
| `BREAKAGE` | 파손 | 재료 결손, 깨짐, 큰 구조 손상 |
| `STEP_CANDIDATE` | 단차 후보 | 높이 차이가 의심되는 경계·들뜸 |

### 변색 처리

`DISCOLORATION`은 평가 MVP에서 독립 YOLO 클래스보다 서버 속성으로 처리한다.

이유:

- 조명 색온도
- 그림자
- 젖음
- 카메라 자동 화이트밸런스
- 주변 바닥 반사
- 촬영 시간

등의 영향이 크기 때문이다.

VLM은 다음을 별도로 출력한다.

- `discoloration_area_ratio`
- `contrast_to_surrounding`
- `color_analysis_confidence`

후속 데이터에서 재현성이 확보되면 독립 클래스로 승격한다.

### 권장 모델

- YOLOv8n-seg
- 입력 640×640
- 2~5 FPS 또는 event-triggered
- 마스크 기반 실제 면적·형상 후처리

## 1.3 모델 C: 서버 VLM

### 모델

- Qwen2.5-VL-7B-Instruct
- Flask 추론 API
- GPU 서버 또는 별도 추론 노드
- 로봇 주행과 비동기

### 역할

- YOLO 클래스 검토
- 결함 상세 설명
- 결함 면적·기능 손실 추정
- RGB 기반 색상 대비·변색 속성
- 하향 ToF 값을 포함한 단차 해석
- 위험도 구성요소 산출
- 추천 조치
- 교체 시기 범주
- 분석 신뢰도

평가일까지는 전체 파인튜닝보다 **구조화 프롬프트 + few-shot 예시 + 규칙 결합**을 우선한다. 파인튜닝은 전문가 검토 라벨이 충분히 쌓인 후 LoRA/QLoRA로 진행한다.

## 2. 데이터 수집 계획

## 2.1 데이터 출처

1. 실제 보도·학교·교육장 주변 점자블록
2. 팀이 제작한 결함 모형
   - 얇은 균열 무늬
   - 표면 벗겨짐
   - 일부 파손
   - 높이가 다른 단차 모형
3. 정상 점자블록의 다양한 재질·색상
4. 하드 네거티브
   - 블록 이음새
   - 그림자
   - 물자국
   - 낙엽
   - 테이프
   - 흙
   - 정상 돌기 홈
   - 점자블록 바깥 보도 크랙
5. 순찰 영상에서 active learning으로 추출한 오탐·미탐 프레임

## 2.2 촬영 조건 매트릭스

| 축 | 최소 조건 |
|---|---|
| 시간 | 오전, 정오, 오후 |
| 조명 | 직사광, 흐림, 그림자, 실내 |
| 표면 | 건조, 약간 젖음 |
| 각도 | 정면, 좌·우 편향, 접근 거리 변화 |
| 재질 | 콘크리트, 고무/합성재, 석재 등 확보 가능한 범위 |
| 상태 | 정상, 경미, 중간, 심각 |
| 주변 | 낙엽, 줄눈, 보도 크랙, 색상 유사 바닥 |
| 속도 | 정지 촬영, 실제 주행 흔들림 |
| 카메라 | 최종 브래킷 높이·각도 중심 |

같은 결함을 연속 영상으로 수백 장 저장하더라도 독립 데이터로 보지 않는다. 물리적으로 다른 위치·결함을 늘리는 것이 우선이다.

## 2.3 현실적인 데이터 목표

평가 전 최소 목표:

| 데이터셋 | 목표 |
|---|---:|
| 점자블록 주행 이미지 | 600~1,000장 |
| 선형·점형 마스크 | 각 400개 이상 |
| 불량 이미지 | 800~1,500장 |
| 각 불량 클래스의 고유 사례 | 최소 150개, 목표 250개 이상 |
| 정상·하드 네거티브 | 전체 불량 이미지의 30~50% |
| 실제 주행 영상 | 루트별 최소 5회 |
| VLM 전문가 검토 샘플 | 우선 100~200건 |

클래스당 고유 사례가 100개 미만이면 클래스를 억지로 유지하지 않는다. 예를 들어 마모와 벗겨짐의 경계가 일관되지 않으면 `SURFACE_WEAR`로 통합한다.

## 3. 라벨링 규칙

## 3.1 위치 단위 분할

Train/Validation/Test는 프레임 단위 무작위 분할이 아니라 **물리 위치 단위**로 나눈다.

권장:

- Train 70%
- Validation 15%
- Test 15%
- 같은 크랙의 연속 프레임은 반드시 같은 split
- 평가 시연 코스의 일부 구간은 최종 test로만 사용

프레임 단위 랜덤 분할은 거의 동일한 이미지가 train과 test에 들어가 정확도가 과대평가되는 문제를 만든다.

## 3.2 점자블록 라벨

- 보이는 블록 영역 전체를 마스크
- 심하게 가려진 영역은 보이는 부분만
- 선형·점형이 동시에 존재하면 각각 별도 마스크
- 블록 사이 정상 줄눈은 결함으로 라벨하지 않음
- 주변 노란색 타일이 점자블록이 아니면 negative

## 3.3 불량 라벨

### `SURFACE_WEAR`

포함:

- 표면 박리
- 돌기 일부 소실
- 넓은 마찰 마모
- 재료층 벗겨짐

제외:

- 단순 얼룩
- 젖음
- 조명 반사
- 정상 제조 질감

### `CRACK`

포함:

- 연속 선형 균열
- 여러 갈래 망상 균열
- 돌기 사이를 가로지르는 균열

제외:

- 블록 경계
- 시멘트 줄눈
- 일정하고 반복적인 제조 홈
- 점자블록 바깥의 보도 크랙

긴 크랙은 하나의 마스크가 이상적이다. 프레임에서 분리되어 보이면 여러 마스크를 허용하고 후처리로 병합한다.

### `BREAKAGE`

포함:

- 블록 일부가 떨어져 나감
- 모서리·돌기 큰 결손
- 내부 재료가 드러난 깨짐
- 블록 구조가 유지되지 않는 손상

크랙과 파손이 함께 있으면 다중 라벨을 허용한다.

### `STEP_CANDIDATE`

포함:

- 블록 들뜸
- 인접 블록 높이 차이가 시각적으로 명확
- 그림자 경계와 표면 변위가 함께 보임

단차는 RGB 라벨만으로 확정하지 않는다. 라벨 메타데이터에 다음을 추가한다.

- 실측 높이 또는 제작 모형 높이
- TF-Luna 변화량
- `confirmed / unconfirmed`
- 촬영 방향

## 4. 라벨 품질관리

### 역할

- 1차 라벨러
- 2차 검수자
- 클래스 경계 분쟁 담당자
- 데이터셋 릴리스 담당자

### 검수 절차

1. 라벨 가이드 예제 30장을 팀 전체가 공동 라벨링
2. 차이가 발생한 기준을 문서화
3. 신규 데이터 10%를 이중 라벨링
4. disagreement가 큰 클래스는 정의 통합
5. 릴리스 전 무작위 100장 검수
6. 데이터셋 버전과 변경 사유 기록

## 5. 학습 방식

## 5.1 초기 학습 설정

권장 시작값:

| 항목 | 설정 |
|---|---|
| 초기 가중치 | COCO pretrained YOLOv8n-seg |
| 입력 크기 | 640 |
| epochs | 80~150, early stopping |
| patience | 20 |
| optimizer | 기본값으로 시작 후 비교 |
| batch | GPU 메모리에 맞춤 |
| mixed precision | 사용 |
| seed | 고정, 예: 42 |
| workers | 환경에 맞춤 |
| class weights | 심한 불균형 시 적용 |
| 모델 선택 | Validation mask mAP와 실제 이벤트 지표를 함께 평가 |

특정 하이퍼파라미터를 초기에 과도하게 탐색하지 않는다. 데이터 품질과 split 누수를 먼저 해결한다.

## 5.2 증강

### 허용

- 밝기·감마
- 색온도·채도 변화
- 약한 blur와 motion blur
- 그림자 합성
- 약한 perspective
- scale·crop
- 노이즈
- 부분 가림
- 수평 flip

### 제한

- vertical flip
- 결함 폭을 비현실적으로 늘리는 강한 perspective
- 크랙 형태를 파손처럼 바꾸는 과한 mosaic
- 실제 단차 높이 의미를 왜곡하는 geometry augmentation
- 변색 데이터의 색 의미를 완전히 제거하는 강한 hue shift

### 증강 검증

증강 이미지 100장을 사람이 직접 확인하고 라벨 의미가 유지되는지 검토한다.

## 6. 평가 지표

## 6.1 점자블록 주행 모델

| 지표 | 목표 |
|---|---:|
| Mask mAP50 | 0.85 이상 목표 |
| Test mask IoU | 0.80 이상 목표 |
| 중심선 평균 횡오차 | 실제 코스에서 5 cm 이내 목표 |
| 10 m 추종 성공률 | 90% 이상 |
| Jetson 추론속도 | 10 FPS 이상 |
| 점자 미검출 오탐 전환 | 정상 구간에서 최소화 |

모델 mAP만으로 주행 성능을 판단하지 않는다. 중심선 오차와 실제 완주율이 최종 지표다.

## 6.2 불량 모델

| 지표 | 목표 |
|---|---:|
| 전체 mask mAP50 | 0.70 이상 목표 |
| 파손 recall | 0.80 이상 목표 |
| 단차 후보 recall | 0.80 이상 목표 |
| 정상 이음새 오탐 | hard-negative test에서 지속 감소 |
| 대표 촬영 성공률 | 결함 통과 10회 중 9회 |
| 동일 결함 중복 이벤트 | 10회 중 1회 이하 |
| Jetson 추론속도 | 2~5 FPS |

클래스별 성능을 따로 공개한다. 소수 클래스가 낮은데 전체 평균만 높은 상황을 허용하지 않는다.

## 6.3 종단 간 이벤트 지표

- 실제 결함 수 대비 생성 이벤트 수
- 한 결함당 이벤트 수
- 잘못된 primary class 비율
- 서버 VLM이 class를 수정한 비율
- 사진이 흐려 분석 불가한 비율
- GPS·루트 거리 누락률
- 온라인/오프라인 업로드 성공률

## 7. 온디바이스 후처리

## 7.1 시간적 다수결

최근 N개 프레임을 누적한다.

```text
S(c) = Σ confidence(c,t) × frame_quality(t) × temporal_weight(t)
```

- `argmax S(c)`를 primary
- 2위가 기준 이상이면 secondary
- 프레임 1장의 결과를 즉시 확정하지 않음
- N은 5~10 프레임에서 실험

## 7.2 트래킹 기반 1회 촬영

ByteTrack, BoT-SORT 또는 IoU+중심거리 기반 단순 추적기를 사용한다.

```text
NEW → TRACKING → CENTERED → CAPTURED → PASSED
```

- `CAPTURED` track은 재촬영 금지
- track이 화면에서 사라지고 충분한 이동거리가 지난 후에만 새 객체 허용
- GPS cooldown만으로 중복을 제거하지 않음

## 7.3 긴 크랙 병합

### 동일 프레임

- 마스크 끝점 거리
- 주축 방향
- 점자블록 영역 내 연결 가능성
- 작은 gap을 closing한 결과

### 연속 프레임

- optical flow 또는 track
- 루트 상 누적거리
- 결함 시작·종료 위치
- 마스크 방향

병합 결과에는 `defect_segment_id`를 부여한다.

## 7.4 크랙·파손 흔들림

- 다중 라벨 허용
- primary/secondary 출력
- 서버 VLM 재검토
- `classification_uncertain=true` 저장
- 지도 대표 아이콘은 위험도가 높은 클래스로 결정

## 7.5 촬영 품질 점수

```text
Q =
0.30 × sharpness
+ 0.20 × exposure
+ 0.20 × mask_size
+ 0.15 × center_alignment
+ 0.15 × low_yaw_rate
```

track의 중앙 진입 전후 프레임 중 Q가 최대인 이미지를 대표 이미지로 선택한다.

- 시스템 역할과 클래스 정의 고정
- 판단 근거를 짧은 항목으로 출력
- JSON 외 텍스트 금지
- 알 수 없는 값을 추정하지 않고 `null`

- 날씨·조명 때문에 색상 분석이 어렵다면 낮은 confidence
- 법적 시설 적합 판정으로 표현하지 않음
- 추천 교체일은 단일 날짜보다 범주형으로 출력

## 9.4 VLM 검증

100~200개의 사람이 검토한 샘플에 대해 다음을 측정한다.

- JSON 파싱 성공률
- 클래스 일치율
- 위험도 구성요소 평균 절대오차
- 위험 구간 순위 상관
- 근거 문장의 이미지 부합 여부
- hallucination 비율
- 동일 이미지 반복 추론 변동

평가 MVP에서 JSON 파싱 실패 시 한 번 재시도하고, 계속 실패하면 규칙 기반 결과로 대체한다.

## 10. 위험도 산출

## 10.1 이벤트 위험도

각 항목은 0~100으로 정규화한다.

```text
R_event =
0.30 × physical_severity
+ 0.25 × tactile_function_loss
+ 0.25 × trip_risk
+ 0.10 × affected_area_score
+ 0.10 × persistence_score
```

### 항목 의미

- `physical_severity`: 파손·균열·박리 정도
- `tactile_function_loss`: 돌기·선형 안내 기능 저하
- `trip_risk`: 단차·날카로운 파손·걸림 가능성
- `affected_area_score`: 결함 길이·면적·블록 수
- `persistence_score`: 여러 순찰에서 반복된 정도

### 신뢰도는 위험도와 분리

```text
confidence =
0.30 × yolo_confidence
+ 0.20 × image_quality
+ 0.20 × vlm_consistency
+ 0.15 × sensor_support
+ 0.15 × repeated_observation
```

위험도가 80이라도 confidence가 낮으면 `고위험 후보·현장 검토 필요`로 표시한다.

## 10.2 클래스별 구성요소

### 마모·벗겨짐

- 돌기 손실 비율
- 박리 면적
- 연속 블록 영향
- 주변 바닥과의 대비

### 크랙

- 실제 길이
- 최대 폭 구간
- 돌기 영역 횡단 여부
- 가지 수와 연결성
- 반복 순찰에서 확대 여부

### 파손

- 결손 면적
- 날카로운 모서리
- 블록 구조 유지 여부
- 인접 블록 확산

### 단차

- ToF 또는 깊이 보조값
- 단차 경계 길이
- 진행 방향을 가로지르는지
- 반복 측정 일관성

## 10.3 위험도 구간

다음은 프로젝트 내부 운영 기준이며 법정 기준이 아니다.

| 점수 | 표시 | 추천 |
|---:|---|---|
| 0~29 | 낮음 | 추세 관찰 |
| 30~49 | 보통 | 다음 정기점검 |
| 50~69 | 주의 | 3~6개월 보수계획 검토 |
| 70~84 | 높음 | 1~3개월 우선 현장 확인 |
| 85~100 | 매우 높음 | 신속한 현장 확인·조치 판단 |

## 11. 구간 위험도와 보수 우선순위

## 11.1 3~5 m 관리 구간

같은 루트의 이벤트를 다음으로 집계한다.

- 루트 ID
- 누적거리 bin
- GPS 근접도
- 결함 종류
- 관측 시간

구간 값:

```text
R_segment =
0.50 × 최대 이벤트 위험도
+ 0.30 × 상위 3개 평균
+ 0.20 × 결함 밀도 점수
```

## 11.2 보수 우선순위

```text
Priority =
R_segment
× route_importance_factor
× deterioration_factor
× confidence_factor
```

예시 factor:

- 일반 구간: 1.0
- 대중교통·복지시설 연결: 1.1~1.3
- 빠른 악화: 1.1~1.4
- 낮은 데이터 신뢰도: 우선순위 숫자를 올리기보다 `현장 검토` 플래그

## 12. 교체 시기 추정 알고리즘

## 12.1 평가 MVP

장기간 시계열이 없으므로 정확한 잔여수명을 학습할 수 없다. 평가 단계에서는 다음 두 요소를 결합한다.

1. 현재 위험도 구간
2. 최근 반복 순찰의 위험도 기울기

관측이 3회 미만이면 규칙 기반 범주만 제공한다.

```text
R < 30  → 다음 정기점검
30~49   → 6~12개월 내 검토
50~69   → 3~6개월 내 검토
70~84   → 1~3개월 내 현장 확인
85 이상 → 신속한 현장 확인
```

## 12.2 시계열 기울기

같은 관리 구간에 관측이 3회 이상 쌓이면 강건 회귀를 사용한다.

```text
R(t) = a + b × t
```

- `b`: 일일 또는 주간 위험도 증가율
- 이상치에 민감한 단순 최소제곱보다 Theil-Sen 또는 Huber regression 검토
- 목표 임계값 `T`까지의 예상 시간:

```text
days_to_threshold = (T - R_now) / b
```

조건:

- `b <= 0`: 교체 시기 계산하지 않고 추세 관찰
- 관측 간격이 너무 짧거나 이미지 품질이 다르면 confidence 감소
- 예상값에 최소·최대 범위 적용
- 예측은 보수 의사결정 참고값임을 표시

## 12.3 예측 신뢰도

```text
forecast_confidence =
관측 수
+ 관측 기간
+ 회귀 잔차
+ 반복 검출 일관성
+ 촬영 품질
```

표시:

- `LOW`: 관측 1~2회 또는 큰 변동
- `MEDIUM`: 관측 3~5회
- `HIGH`: 충분한 기간과 안정적 추세

평가일까지는 대부분 `LOW` 또는 `MEDIUM`이 정상이다.

## 12.4 후속 파인튜닝

실제 지자체 보수 이력과 전문가 판정을 확보한 후 다음을 학습한다.

- 입력: 이미지 특징, 결함 면적, 단차, 재질, 기후, 교통·보행 영향, 과거 위험도
- 출력: 다음 검사 시 위험도, 임계값 도달 기간, 보수 방식
- 방법: gradient boosting 또는 survival model부터 시작
- 충분한 데이터 전에는 복잡한 딥러닝 시계열 모델을 사용하지 않음

## 15. 일정 지연 시 AI 범위 축소

1. 점자 주행 모델의 두 클래스를 하나의 `TACTILE_BLOCK` 마스크로 통합
2. 마모·벗겨짐을 단일 클래스 유지
3. 변색 독립 클래스 개발 중단
4. 단차를 카메라 후보 + 센서 플래그로만 표시
5. VLM 파인튜닝을 제외하고 few-shot 프롬프트 사용
6. 교체 시기를 규칙 기반 범주로 제한
7. TensorRT engine 문제가 있으면 검증된 ONNX Runtime 배포

정확도 숫자를 높이기 위해 test split을 오염시키는 방식은 허용하지 않는다.
